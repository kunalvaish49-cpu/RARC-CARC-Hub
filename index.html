NDC Does Not Correspond to HCPCS/CPT Billed

Drug Billing / Data Mismatch·Professional & Facility Claims
Common CARC Pairing
CO-16 CMS-1500 UB-04
What This Means

The National Drug Code (NDC) reported on the claim does not match the HCPCS/CPT drug code (typically a J-code) billed on the same service line.

ℹ️
Important Distinction
N846 does not automatically mean the drug is not covered or the claim is permanently denied. It means the NDC and J-code need to match up before processing can continue.

N4 Qualifier Format — Common Sub-Error
The N4 qualifier must appear as a single unspaced prefix directly before the 11-digit NDC. A space between N4 and the NDC number will cause a format rejection even if the NDC itself is correct.
Why This Happens
Data Entry & Selection Issues
  • Wrong NDC selected for the drug actually administered or NDC from a different manufacturer was used
  • A 10-digit NDC was not converted to the required 11-digit (5-4-2) format
  • The HCPCS/J-code does not align with the NDC submitted
  • NDC is expired, discontinued, or no longer active in the payer's drug file

How to Fix It
4a. Standard Correction Process
  1. 1Pull the MAR and vial or package label from the date of service
  2. 2Confirm the physical form of the drug — powder, liquid, cream, biologic
  3. 3Verify the quantity aligns with the UOM selected (e.g., if ML — bill actual mLs administered, not vial size)
  4. 4Confirm the corrected UOM is consistent with the billed J-code and NDC and Submit a corrected claim with Resubmission Code 7
4b. Discontinued or Inactive NDC
  • Check the FDA NDC database or drug wholesaler records for a valid replacement NDC from the same manufacturer for the same drug
  • If the drug has been reformulated or the manufacturer has changed, identify the NDC that was active on the date of service
  • If no valid NDC can be confirmed for the DOS, escalate to coding — a NOC (not otherwise classified) code may be required with supporting documentation

What NOT to Do

  • ✅Claim run through edits before submission

N816 SUPPLEMENTAL CO-16

Invalid NDC Unit of Measure

Drug Billing / Unit of Measure Error · Professional & Facility Claims
Common CARC Pairing
CO-16 CMS-1500 UB-04
What This Means

The Unit of Measure (UOM) reported alongside the NDC on the claim is either missing, invalid, or does not match how the drug was supplied and administered. Payers use the UOM to verify that the quantity billed is logical for the drug form on file. When it doesn't match, the claim is flagged and suspended until corrected.

ℹ️
Important Distinction
N816 does not mean the drug is not covered. It means the UOM field needs to be corrected before the claim can process. In most cases this is a straightforward corrected claim — not an appeal.

Why This Happens
Most Common Causes

  • Wrong UOM selected — e.g., billing UN for a liquid drug that should be ML
  • UOM does not match the drug form in the payer's NDC drug file
  • Drug reformulated or repackaged and the UOM changed with it
  • Copied from a prior claim where a different drug form was used
  • Compound or multi-component drugs billed without clarifying the UOM per component

How to Fix It
5a. Standard Correction (Most Cases)
  1. 1Pull the MAR and vial or package label from the date of service
  2. 2Confirm the physical form of the drug — powder, liquid, cream, biologic
  3. 3Match the drug form to the correct UOM using the table above
  4. 4Verify quantity aligns with UOM selected (e.g., if ML — bill actual mLs administered, not vial size)
  5. 5Confirm corrected UOM is consistent with the billed J-code and NDC
  6. 6Submit a corrected claim with Resubmission Code 7
5b. Drug Reformulated or Repackaged
  • Check the current vial or package label for the updated drug form
  • Do not carry forward UOM from a prior claim — always verify from the current label
  • If the NDC itself changed due to reformulation, verify the new NDC maps correctly to the J-code before resubmitting
5c. Compound Drugs
  • Each component requires its own NDC, UOM, and quantity reported separately
  • Escalate to coding if there is any ambiguity about which UOM applies to which component
⚠️
For compound drugs, never apply a single UOM across all components — this will cause N816 on every component line.

✅ Appeal Only When ALL of These Are True

  • The UOM on the corrected claim is confirmed correct against the MAR and vial label
  • The continued denial is due to a payer-side system or drug file mapping error

🔄 Correct & Resubmit First (Most Cases)

  • The large majority of N816 denials resolve by correcting the UOM and resubmitting
  • Do not appeal as a first step — verify drug form, UOM code, and quantity before resubmission
  • Use Resubmission Code 7 on the corrected claim

M119 REMARK CO-16

Missing / Incomplete / Invalid / Deactivated / Withdrawn NDC

Drug Billing / NDC Data Issue · Professional & Facility Claims
Common CARC Pairing
CO-16 CMS-1500 UB-04
What This Means

M119 is a remark code — not a RARC — used by payers to flag that the National Drug Code (NDC) on the claim is one of the following: completely missing, incomplete (e.g., wrong digit count), formatted incorrectly, or the NDC has been deactivated or withdrawn from the FDA drug database. The payer cannot process the drug line until a valid, active NDC is present.

⚠️
Critical — Identify the Issue Type First
M119 covers four distinct problem types. Before taking any action, you must identify which of the four applies — each has a different resolution path. A missing NDC is a data entry fix. A withdrawn NDC may require a NOC code and clinical documentation. Treating all M119 denials the same way is the most common mistake.

DeactivatedNDC existed but FDA or manufacturer has deactivated it — drug may still be available under a new NDCLocate the replacement active NDC for the same drug and DOS
WithdrawnNDC has been recalled or pulled from the market by FDA or manufacturerConfirm whether a NOC code applies; escalate to coding

How to Fix It — By Issue Type
4a. Missing NDC
  1. 1Pull the MAR and vial or package label from the date of service
  2. 2Locate the 11-digit NDC from the label — use the actual administered vial, not a reference sheet
  3. 3Confirm UOM and quantity match the drug form and MAR documentation
  4. 4Validate NDC maps correctly to the billed J-code using a crosswalk
  5. 5Submit a corrected claim with Resubmission Code 7
4b. Incomplete or Invalid NDC
  • Pull the original claim and identify exactly what was submitted in the NDC field
  • Compare against the vial label or FDA NDC database
  • Check digit count (must be exactly 11), format (5-4-2), N4 qualifier (no space), and dashes (confirm payer EDI guide)
  • Correct the NDC field and resubmit with Resubmission Code 7
4c. Deactivated NDC
  • Confirm the NDC is deactivated via FDA NDC database or wholesaler records
  • Identify the active replacement NDC for the same drug, strength, and form on the DOS
  • Verify the replacement NDC was active and available on the actual DOS — do not use an NDC that became active after the DOS
  • Validate the replacement NDC maps to the same J-code
  • If a valid active NDC exists for DOS → correct and resubmit
  • If no active NDC existed on DOS → escalate to coding for NOC code evaluation
4d. Withdrawn NDC
  • Confirm the withdrawal via FDA records or manufacturer notice
  • If drug was legitimately administered before the withdrawal was known → escalate to coding — a NOC code with supporting documentation is likely required
  • If drug should not have been administered → coordinate with clinical and compliance teams before billing
NOC (Not Otherwise Classified) Code Path

When no valid NDC can be identified for the DOS — typically in deactivated or withdrawn scenarios — the claim may need to be billed using a NOC drug code. This is a coding-level decision, not an AR-level decision.

Escalate to Coding When
  • NDC was deactivated before DOS and no replacement NDC existed at that time
  • NDC was withdrawn/recalled and no substitute NDC applies
  • Drug is compounded with no individual component NDCs available
  • A biosimilar was administered and NDC mapping is unclear
  • Payer's drug file has no record of the NDC at all, even though it appears valid in FDA database
When a NOC Code Is Billed
  • J-code is replaced with the appropriate NOC code (e.g., J3490 — unclassified drugs; J3590 — unclassified biologics)
  • A complete drug description must accompany the NOC code: drug name, strength, dosage form, and route of administration
  • Some payers require prior authorization or letter of medical necessity alongside a NOC claim
🚫
Do Not Bill NOC Without Coding Validation
Never submit a NOC code without coding team sign-off and a proper drug description. Incorrect NOC billing will result in another denial and consumes timely filing days.

N60 REMARK CO-16

A valid NDC is required for payment of drug claims. Effective October 2.

Drug Billing / NDC Compliance Requirement · Professional & Facility Claims
Common CARC Pairing
CO-16 CMS-1500 UB-04
What This Means

N60 is a payer-generated remark code indicating that a valid NDC is required on the claim in order to process payment for a drug service — and that requirement has been in effect since October 2 of the applicable year. This is a compliance-driven denial, not a data entry error. The payer is telling you that your billing workflow has not yet been updated to include a mandatory NDC on drug lines.

Key Distinction: N60 vs. M119
CodeWhen It Fires
M119NDC was submitted but is missing, incomplete, invalid, deactivated, or withdrawn
N60No valid NDC was submitted at all, or payer's system cannot recognize the NDC as meeting the new mandatory requirement
💡
N60 Does Not Mean the Drug Is Not Covered
N60 means the claim cannot be paid without a properly submitted, valid NDC. In most cases this resolves with a corrected claim. N60 is often seen in bulk on the first wave of denials after a payer activates NDC enforcement — meaning it may affect multiple claims across multiple dates of service simultaneously.
Why N60 Happens
Scenario 1 — New Payer NDC Enforcement

The payer activated a policy (effective October 2) requiring NDCs on drug claims. Your billing system or workflow was not updated in time, so claims submitted without an NDC are being denied in bulk.

Scenario 2 — Individual Claim NDC Gap

A specific drug claim was submitted without an NDC, or with an NDC the payer cannot validate (wrong format, wrong qualifier, wrong segment placement), even though NDC reporting was otherwise in place.

Most Common Root Causes
  • NDC field was blank — drug was billed by J-code only, no NDC attached
  • NDC submitted but N4 qualifier missing or incorrectly formatted
  • 10-digit NDC submitted instead of required 11-digit (5-4-2) format
  • NDC placed in the wrong EDI loop or claim field
  • Billing system default not updated after payer began enforcing NDC requirement
  • Charge entry template missing the NDC capture field
  • Drug purchased from a new supplier and NDC not captured at point of administration

N815 REMARK CO-16

Missing / Incomplete / Invalid NDC Unit Count

Drug Billing / NDC Unit Count Error · Professional & Facility Claims
Common CARC Pairing
CO-16 CMS-1500 UB-04
What This Means

N815 is a payer remark code indicating that the NDC unit count (quantity) on the drug claim line is missing, incomplete, or invalid. Unlike N816 which targets the Unit of Measure code itself, N815 fires when the numeric quantity value associated with the NDC is absent, zero, illegible, or does not make clinical or billing sense for the drug and UOM submitted. The payer cannot calculate payment without a valid administered quantity.

N815 vs. N816 — Key Distinction
CodeWhat It TargetsExample Issue
N815The quantity / unit count value is missing, zero, or invalidNDC submitted with UOM of ML but quantity field is blank or 0
N816The UOM code itself is missing, wrong, or unrecognizedBilling GR for a liquid drug that should be ML
💡
N815 Does Not Mean the Drug Is Not Covered
N815 means the claim cannot be priced or paid because the administered quantity is not readable by the payer's system. In nearly all cases this resolves with a corrected claim containing the verified quantity from the MAR.
Why N815 Happens
Most Common Root Causes
  • NDC quantity field left blank or populated with a zero at charge entry
  • Billing system default populated the quantity field with a placeholder (e.g., "1" for a multi-mL dose) without clinical verification
  • Quantity not captured at point of administration — biller estimated rather than documented from MAR
  • Drug vial size billed instead of actual quantity administered (e.g., billing full 10 mL vial when only 4 mL were given)
  • Compound drug with multiple components — quantity reported for the compound rather than per component
  • JW line submitted without a quantity on the primary administered line
  • NDC line formatted correctly but quantity segment truncated or missing in the EDI 837 transaction
Quantity vs. Vial Size — Critical Rule
⚠️
Always bill what was administered — not what was in the vial.

If a 10 mL vial was used but only 6 mL were administered, the NDC quantity must be 6 (plus a JW line for the 4 mL discarded, if applicable). Billing the vial size instead of the administered quantity is both an N815 trigger and a compliance issue.

  1. 1Pull the MAR for the date of service — the quantity must come from the MAR, not the vial size or a prior claim

  3. 3Confirm the quantity is greater than zero and reflects only what was administered to the patient
  4. 4Confirm the UOM code is correct for the drug form — if both quantity and UOM are wrong, fix both

  7. 7Submit corrected claim with Resubmission Code 7
5b. JW / Wastage Quantity Rules
💊

  • Primary line: quantity = actual mLs (or units) administered
  • JW line: quantity = actual mLs (or units) discarded — same UOM as primary
  • Primary quantity must be greater than zero — a JW-only line with no administered line will deny
  • If no drug was wasted, use JZ modifier (required by some payers) to affirm zero wastage
5c. Compound Drugs
  • Each component requires its own NDC, UOM, and quantity reported separately
  • Do not report the total compound quantity against a single NDC — this will cause N815 on every component line
  • Escalate to coding if there is any ambiguity about the administered quantity per component

N847 REMARK CO-16

National Drug Code (NDC) Billed Is Obsolete

Drug Billing / Obsolete NDC · Professional & Facility Claims
Common CARC Pairing
CO-16 CMS-1500 UB-04
What This Means

N847 is a payer remark code indicating that the National Drug Code (NDC) submitted on the claim has been identified as obsolete — meaning the NDC existed at one point but has since been retired, replaced, or superseded in the payer's drug file or the FDA NDC database. An obsolete NDC is different from a deactivated or withdrawn NDC in that the drug may still be available under a new or updated NDC, often because the manufacturer changed packaging, labeling, or product formulation. The payer cannot process payment against an obsolete NDC.

N847 vs. Related NDC Denial Codes — Key Distinctions

N846NDC does not correspond to the HCPCS/J-code billedCorrect the NDC or J-code to match each other via crosswalk
N60No valid NDC was submitted at all — payer NDC mandate not metAdd a valid NDC to the claim and resubmit
⚠️
Obsolete ≠ Drug Is Not Covered
N847 does not mean the drug is not covered or the claim is permanently denied. It means the specific NDC number submitted is no longer recognized. In most cases, the same drug is available under a current, active NDC that the payer will accept. The correction is to identify and submit the correct NDC that was active and in use on the date of service.
Why N847 Happens
Most Common Root Causes
  • NDC was copied from a prior claim where a different lot, package size, or manufacturer version was used — and that NDC has since been retired
  • Drug manufacturer changed packaging or labeling, issuing a new NDC while retiring the old one — billing system was not updated
  • Charge entry template or billing system default still carries a legacy NDC that is no longer current
  • Drug was purchased from a new wholesaler or distributor whose product carries a different, current NDC than the legacy NDC in the system
  • NDC was entered from the outer carton or a product insert rather than the actual administered vial label — carton NDC may be obsolete while vial NDC is current
  • Biosimilar or generic substitution occurred — NDC from the original reference product was billed instead of the administered biosimilar/generic
  • Payer NDC drug file update cycle caused a previously accepted NDC to be flagged as obsolete upon a policy refresh
How NDCs Become Obsolete
💡
The FDA NDC database is updated continuously. When a manufacturer repackages, reformulates, or changes labeling, the old NDC is retired and a new one is issued. The drug may be identical — only the NDC changes. Payer drug files mirror these updates, so an NDC that cleared last quarter may be flagged N847 after the next payer file refresh.

National Drug Code (NDC) Billed Cannot Be Associated With a Product

Drug Billing / NDC Product Association Error · Professional & Facility Claims
Common CARC Pairing
CO-16 CMS-1500 UB-04
What This Means

N848 is a payer remark code indicating that the NDC submitted on the claim cannot be linked to any identifiable drug product in the payer's drug file or the FDA NDC database. Unlike N847 (where the NDC existed but is now obsolete) or M119.

N848 vs. Related NDC Denial Codes — Key Distinctions

⚠️
N848 Is Not Always a Clinical Error — Check Data Entry First
The most common cause of N848 is a digit transposition, a missing digit, or an extra digit in the NDC number — creating a string that does not match any real product. Always verify the NDC character by character against the actual vial label before assuming the drug has a product registration problem.
Why N848 Happens
Data Entry & Format Errors
  • Digit transposition — e.g., 12345678901 entered as 12354678910 — creates an NDC that resolves to no product
  • Missing digit — 10-digit NDC submitted without converting to 11-digit (5-4-2), and the result does not match any product
  • Extra digit typed — 12-digit string submitted, which fails all lookup matches
  • Dashes included in the NDC string in a field that requires a dash-free 11-digit format — or vice versa
Product Registration & System Issues
  • Drug was compounded — compounded preparations typically do not have FDA-registered NDCs and will not resolve in a payer drug file
  • Investigational or clinical trial drug — NDC may not be in the commercial drug file
  • Drug imported or sourced outside standard US distribution channels — NDC may not be registered with FDA
  • Payer drug file gap — NDC is valid in the FDA database but has not been loaded into the payer's pricing system
💡
Key test: Before taking any correction action, enter the NDC into the FDA OpenFDA drug database or DailyMed. If it returns no results, the NDC itself is the problem — not the payer's file. If it returns a valid product, the issue is on the payer side.

How to Fix It
7a. Data Entry Error — Single Claim Correction
  1. 1Pull the actual vial or package label from the date of service
  2. 2Compare the billed NDC digit by digit against the label — identify the transposition, missing digit, or extra digit
  3. 3Enter the NDC from the vial label into the FDA OpenFDA database or DailyMed to confirm it resolves to a real product
  4. 4Convert 10-digit NDC to 11-digit (5-4-2) format if needed
  5. 5Verify N4 qualifier is attached with no space before the corrected NDC
  6. 6Confirm the corrected NDC maps to the billed J-code using a current crosswalk
  7. 7Verify UOM and quantity are correct per the MAR
  8. 8Submit corrected claim with Resubmission Code 7
7b. NDC Resolves in FDA Database but Payer Cannot Associate It
💡
If the NDC is confirmed valid in the FDA database but the payer returns N848, the payer's drug pricing file has not been updated to include this product. This is a payer-side issue.
  • Call the payer and confirm whether the NDC is loaded in their drug pricing file
  • Request the payer add the NDC to their drug file or provide an accepted alternate NDC for the same product
  • Document the call with rep name, reference number, and outcome
  • If payer confirms the NDC is valid but their system cannot associate it — appeal with FDA product confirmation attached
7c. Compounded Drug or No FDA-Registered NDC
  • Compounded preparations and investigational drugs typically do not have FDA-registered NDCs
  • Escalate to coding immediately — a NOC (not otherwise classified) code such as J3490 or J3590 may be required with supporting clinical documentation
  • Do not resubmit with a fabricated or estimated NDC — this will not resolve N848 and may create a compliance issue

Payer-Specific NDC Product Association Requirements

Drug file loading and NDC product association vary significantly by payer. Verify before submitting corrected claims:

Payer TypeKey Guidance
Medicare (Part B)NDC must resolve to a product in the MAC drug pricing file (ASP-based pricing). If the NDC is valid in the FDA database but not in the MAC drug file, contact the MAC for guidance. Some drugs require a specific NDC tied to ASP pricing — refer to MLN Matters articles and MAC bulletins for the drug in question.
MedicaidState Medicaid drug files are maintained independently. A valid FDA NDC may still return N848 if the state has not loaded the product. Check the state Medicaid drug rebate file and billing manual. Some states have a drug exclusion list — confirm the drug is covered before correcting.
Commercial / Managed CareCommercial payer drug files vary widely in update frequency and product coverage. N848 from a commercial payer frequently indicates a drug file loading lag for a newer or niche product. Call the payer's provider relations or pharmacy team to confirm whether the NDC needs to be loaded before resubmitting.
Medicare AdvantageFollows Medicare Part B drug file in most cases but MA plans maintain their own formulary and drug pricing files. An NDC that resolves at the MAC level may still return N848 at the MA plan level. Confirm with the MA plan's pharmacy or claims department before resubmitting.
What NOT to Do
  • Do not resubmit with the same unresolvable NDC — the claim will deny again with N848
  • Do not fabricate or estimate an NDC — submitting a made-up NDC that happens to resolve to a different product creates a false claim risk
  • Do not use a different drug's NDC as a substitute without clinical confirmation that the same drug was administered
  • Do not bypass coding escalation for compounded or investigational drugs — a NOC code path is required and cannot be resolved at the AR level
  • Do not appeal before first verifying the NDC in the FDA database and attempting a corrected claim with the verified NDC
  • Do not carry the unresolvable NDC to secondary or tertiary claims — correct it on all payer submissions
🚫
Submitting any NDC that does not resolve to the actual drug administered — even if it passes the payer's system — is a compliance and false claim risk. Always verify the NDC resolves to the correct product before resubmitting.
Prevention Checklist

Before submitting any drug claim, confirm:

  • ✅NDC captured character by character from the actual administered vial label — not typed from memory or copied from a prior claim
  • ✅NDC verified in FDA OpenFDA or DailyMed database — confirms it resolves to a real, identifiable product
  • ✅NDC is exactly 11 digits in 5-4-2 format — no missing, extra, or transposed digits
  • ✅N4 qualifier present with no space before the NDC
  • ✅NDC resolves to the same product as the billed J-code via current crosswalk
  • ✅Charge entry template does not contain a placeholder, test, or legacy NDC as a default
  • ✅For compounded or investigational drugs — coding team consulted before submission; NOC code path confirmed if applicable
  • ✅Drug invoice on file to confirm NDC matches the product received and administered
  • ✅UOM and quantity correct per MAR documentation
  • ✅Claim run through edits before submission
Best Practice — When to Appeal vs. When to Correct

✅ Appeal Only When ALL of These Are True

  • A corrected claim with the verified NDC has already been submitted and denied again with continued N848
  • The FDA OpenFDA or DailyMed database confirms the NDC resolves to a real, identifiable drug product
  • The continued denial is due to a payer-side drug file gap — the product has not been loaded into the payer's pricing system
  • Payer contact has been documented confirming the NDC is valid but not in their file

🔄 Correct & Resubmit First (Most Cases)

  • Most N848 denials resolve by correcting a digit transposition, format error, or placeholder NDC and resubmitting with the verified product NDC
  • Do not appeal as a first step — verify the NDC in the FDA database, fix the data entry error, confirm the J-code crosswalk
  • Escalate to coding before resubmitting if the drug is compounded or investigational
  • Use Resubmission Code 7 on the corrected claim

MA30 REMARK CO-16

Missing / Incomplete / Invalid Type of Bill

Facility Billing / UB-04 · Type of Bill (FL4) · Inpatient & Outpatient
Common CARC Pairing
CO-16 UB-04 FL4 Facility Claims
What This Means

The Type of Bill (TOB) field on the UB-04 claim (Form Locator 4) was either left blank, submitted with an invalid code, or contains an incomplete value that does not meet payer requirements. The TOB is a critical 3-digit (or 4-character with leading zero) code that tells the payer exactly what kind of facility submitted the claim, what type of care was rendered, and whether this is an opening, interim, or closing bill in a series — will suspend or deny it with MA30.

ℹ️
Important Distinction
MA30 Is Always a UB-04 Denial — Never CMS-1500 — MA30 does not mean the services are not covered. It means the claim cannot be processed until a valid, complete, and internally consistent Type of Bill is present in FL4. In the vast majority of cases this is a corrected claim — not an appeal.

How to Read Type of Bill (TOB) — Beginner Friendly Explanation

Type of Bill (TOB) tells the payer three important things:

  • What type of facility billed the claim
  • What kind of care the patient received
  • What kind of claim is being submitted (original, corrected, void, etc.)
Simple Formula:
TOB = Facility Type + Care Type + Claim Frequency

Why MA30 Happens — Root Causes by Category
Missing TOB
  • FL4 was left completely blank during claim entry
  • Billing system failed to populate FL4 due to a mapping or interface error between the EHR and the billing system
  • Claim template was missing the TOB field or the field was suppressed
  • EDI 837I transaction did not include the CLM05 loop where the TOB is transmitted
Incomplete TOB
  • Only 2 or 3 digits submitted — the frequency code (4th digit) is the most commonly omitted digit
  • TOB submitted as 3 characters instead of 4 — leading zero dropped
  • EDI transaction truncated the TOB field
Invalid TOB — Wrong Facility Type
  • Facility type digit does not match the provider's enrollment type — e.g., an ASC billing with a hospital TOB (013X) instead of an ASC TOB (085X)
  • SNF billing with a hospital TOB instead of the correct SNF TOB (021X)
  • HHA billing outpatient TOB instead of home health TOB (032X)
  • Hospice billing with a hospital outpatient TOB instead of the correct hospice TOB (081X or 082X)
Invalid TOB — Wrong Bill Classification
  • Outpatient services billed with inpatient classification — e.g., 0111 for an outpatient infusion that should be 0131
  • Inpatient Part A billed with Part B classification — or vice versa — causing cross-coverage denial
  • Swing bed claims billed with standard inpatient TOB instead of swing bed classification
  • Part B drugs billed to inpatient Part A claim instead of a separate 012X claim
Invalid TOB — Wrong Frequency Code
  • Admit-through-discharge claim (frequency 1) submitted when the patient was never discharged — creating a false episode closure
  • Interim first claim (frequency 2) submitted when there are no prior claims but the patient has since been discharged — should be frequency 1
  • Interim continuing claim (frequency 3) submitted without a prior frequency 2 claim in the system — payer cannot find the first bill
  • Late charge claim (frequency 5) submitted when the original claim has not yet been paid
  • Replacement claim (frequency 7) submitted without a valid prior claim to replace
  • Void claim (frequency 8) submitted against a claim that was never paid or processed
  • Home health final claim (frequency 9) submitted when the episode has not been opened with a prior RAP or initial claim
System & Interface Errors
  • Billing system updated with new TOB values but old TOBs carried forward in templates
  • Crosswalk between EHR care setting and TOB not updated after a facility added a new service line
  • Manual override of TOB during claim scrubbing introduced an invalid combination
FL4 — Where to Report the Type of Bill on the UB-04
UB-04 Form Locator 4 (FL4)
  • Field: FL4 — Type of Bill
  • Format: 4-character alphanumeric — always begins with "0"
  • Position: Upper left section of the UB-04, directly below FL3 (Medical Record Number) and FL2 (Patient Control Number)
  • Required on: All UB-04 institutional claims — inpatient, outpatient, SNF, HHA, hospice, RHC, FQHC, ASC
  • EDI 837I placement: Loop 2300, CLM05-1 (Facility Code Value), CLM05-2 (Facility Code Qualifier), CLM05-3 (Claim Frequency Type Code)
💡
In the EDI 837I transaction, the TOB is split across the CLM05 composite element. CLM05-1 carries the facility type and classification, CLM05-2 carries the qualifier "B" (indicating institutional), and CLM05-3 carries the frequency code. A mapping error in any of these three sub-elements will cause MA30.
TOB Format Rules
  • Always exactly 4 characters — never 3, never 5
  • Always begins with "0" in position 1 — this is non-negotiable
  • Position 2 must be a valid facility type code (1–8)
  • Position 3 must be a valid classification code consistent with the facility type in position 2
  • Position 4 must be a valid frequency code (0–9) consistent with the claim's billing sequence
  • The combination of positions 2, 3, and 4 must be a recognized and payer-accepted TOB combination — not all mathematical combinations are valid

How to Fix MA30 — Step-by-Step Correction Workflow
Step 1 — Identify the Specific MA30 Sub-Issue

MA30 covers three distinct problems. Before correcting, determine which applies:

Sub-IssueHow to IdentifyFirst Action
Missing TOBFL4 is blank on the original claim — pull the 835/ERA or claim copy to confirmDetermine correct TOB from facility type, care setting, and episode status
Incomplete TOBFL4 shows only 2 or 3 characters — frequency digit most commonly missingAdd the missing digit(s) — verify frequency code matches the claim's position in the billing sequence
Invalid TOBFL4 has 4 characters but wrong facility type, classification, or frequency for this claimIdentify which digit is wrong and determine the correct value from the tables above
Step 2 — Determine the Correct TOB
  1. 1Confirm the facility type — what kind of facility submitted this claim? (Hospital, SNF, HHA, Hospice, RHC, ASC, etc.)
  2. 2Confirm the care setting / bill classification — was this inpatient Part A, inpatient Part B, outpatient, swing bed, or another classification?
  3. 3Confirm the frequency — is this patient admitted and discharged (frequency 1), still active (frequency 2, 3), being closed out (frequency 4), a late charge (frequency 5), a replacement (frequency 7), or a void (frequency 8)?
  4. 4Assemble the 4-digit TOB: 0 + facility type + classification + frequency
  5. 5Verify the assembled TOB against your payer's list of accepted TOB combinations — not every mathematically valid string is payer-accepted
Step 3 — Correct and Resubmit
  1. 1Update FL4 with the correct 4-digit TOB in your billing system
  2. 2Confirm the corrected TOB is consistent with all other claim fields — revenue codes, occurrence codes, value codes, and condition codes must all align with the TOB
  3. 3If the TOB change affects the claim type (e.g., changing from outpatient to inpatient), review all associated fields for consistency — a TOB change may cascade into revenue code, occurrence code, and diagnosis code changes
  4. 4Run the corrected claim through your claim scrubber before submission
  5. 5Submit the corrected claim — use frequency code 7 (replacement) in the TOB if correcting a previously submitted claim, or frequency 1 if submitting for the first time
Step 4 — Special Scenarios
⚠️
Changing from Outpatient to Inpatient TOB
If the MA30 correction requires changing the bill classification from outpatient to inpatient (or vice versa), this is not a simple field correction. It changes the entire payment system, benefit category, and claims processing pathway. Escalate to your billing supervisor and confirm with the facility's compliance team before resubmitting. An incorrect TOB that overstates the level of care may create overpayment liability.
⚠️
Interim Billing Sequence Errors
If the MA30 is due to a frequency code error in an interim billing series (e.g., a frequency 3 submitted without a prior frequency 2), you may need to submit or correct multiple claims in sequence before the payer can process the denied claim. Contact the payer to confirm the correct remediation path — correcting a single claim in isolation may not be sufficient.

Payer-Specific TOB Requirements

TOB rules and edits vary significantly by payer and program. Verify before resubmitting:

Payer TypeKey Guidance
Medicare (FFS — MAC)CMS publishes the definitive TOB code list in the Medicare Claims Processing Manual (Pub. 100-04). MAC FISS system applies strict TOB edits — an invalid TOB will suspend or reject the claim before it reaches a human reviewer. TOB drives the applicable PRICER (IPPS, OPPS, SNF PDPM, HHA PDGM, Hospice per diem) — the wrong TOB results in the wrong payment system being applied, not just a denial. Condition code D4 is required when resubmitting a corrected claim to replace a previously processed claim.
Medicare Advantage (MA Plans)MA plans follow CMS TOB structure but may apply additional plan-specific edits. Some MA plans have narrower accepted TOB lists — confirm with the plan's provider manual before submitting. MA plans may also apply prior authorization requirements that are TOB-specific — a TOB change may trigger a retro-auth review.
Medicaid (State Programs)State Medicaid programs follow the UB-04 TOB structure but may have state-specific TOB restrictions. Some states do not accept certain facility type codes from specific provider types. Check the state Medicaid billing manual and companion guide — state edits may be more restrictive than CMS baseline. Some states require a prior authorization number that is linked to the TOB — a TOB change may invalidate the auth.
Commercial / Managed CareCommercial payers use the UB-04 TOB but may have plan-specific accepted TOB lists, especially for carved-out benefit categories (behavioral health, substance use, home health). Confirm with the payer's EDI companion guide — some commercial payers do not accept interim billing frequencies (2, 3, 4) and require admit-through-discharge claims only.
TRICARE / VAFollows CMS TOB structure for institutional claims but has specific rules for military treatment facility (MTF) coordination. Confirm TOB with the TRICARE regional contractor or VA Fee Basis program before resubmitting.
Workers CompensationWC payers vary significantly — many accept UB-04 with standard TOBs but some state WC programs have modified TOB requirements. Confirm with the WC carrier or state program before resubmitting an MA30-denied claim.
What NOT to Do
  • Do not resubmit the claim with the same invalid or incomplete TOB — it will deny again with MA30
  • Do not change the TOB without reviewing all cross-validated fields — a TOB correction that is not reconciled with revenue codes, admission date, discharge status, and occurrence codes will generate secondary denials
  • Do not change the bill classification from outpatient to inpatient (or vice versa) without supervisor and compliance review — this changes the entire payment pathway and may create overpayment liability
  • Do not use frequency code 7 (replacement) without a valid prior claim to replace — submitting a replacement against a claim that was never processed will create a new claim rather than replace the denied one
  • Do not use frequency code 8 (void) unless the original claim was paid and must be fully reversed — a void against an unpaid claim may not process correctly
  • Do not appeal MA30 as a first step — MA30 is almost always a correctable billing error, not a coverage or medical necessity issue
  • Do not carry an incorrect TOB to secondary or tertiary claims — the wrong TOB on the primary will generate a cascade of secondary denials
  • Do not submit interim billing frequency codes (2, 3, 4) without confirming the payer accepts interim billing — some commercial payers require admit-through-discharge claims only
🚫
Appealing MA30 before correcting the TOB error wastes timely filing days and is almost never successful — the payer will uphold the denial because the claim data itself is incorrect. Correct the TOB, reconcile all cross-validated fields, and resubmit before considering an appeal.
Prevention Checklist

Before submitting any UB-04 institutional claim, confirm:

  • ✅FL4 is populated — not blank, not truncated
  • ✅TOB is exactly 4 characters and begins with "0"
  • ✅Position 2 (facility type) matches the provider's enrollment type and care setting
  • ✅Position 3 (bill classification) is consistent with the level of care and benefit category
  • ✅Position 4 (frequency) accurately reflects where this claim falls in the billing sequence
  • ✅The full 4-digit TOB combination is a recognized and payer-accepted code — not just mathematically valid
  • ✅Revenue codes in FL42 are consistent with the TOB
  • ✅Admission date (FL12) is populated if TOB is inpatient or SNF
  • ✅Discharge status (FL17) is populated and consistent with the frequency code
  • ✅Condition code D4 is included when resubmitting a corrected claim (frequency 7)
  • ✅For interim billing series — prior claims in the sequence are confirmed submitted and received before sending the next interim bill
  • ✅Billing system TOB defaults reviewed and updated when new service lines or care settings are added to the facility
  • ✅Claim run through a scrubber with TOB validation before submission
Best Practice — When to Appeal vs. When to Correct

✅ Appeal Only When ALL of These Are True

  • A corrected claim with the verified TOB has already been submitted and denied again
  • The TOB on the corrected claim is confirmed correct per NUBC guidelines and the provider's CCN type
  • The continued denial with MA30 is due to a payer-side system mapping error or provider enrollment mismatch
  • You have documentation confirming the TOB is valid for this facility type, care setting, and billing period

🔄 Correct & Resubmit First (Overwhelming Majority of Cases)

  • Nearly all MA30 denials are billing errors — missing, 3-digit instead of 4-digit, or wrong facility/classification/frequency digit
  • Do not appeal as a first step — identify the specific TOB error, correct it, and resubmit with Resubmission Code 7
  • If the error is a replacement or void without ICN, add the original claim ICN to FL64 and resubmit
  • Contact the payer only if the corrected claim denies again with no change in denial reason
